## Решение #1. Повторный запуск = clean-reinstall по умолчанию

**Дата**: 2026-04-19 (по обратной связи от первого живого клиента)
**Вопрос**: как вести себя если клиент запускает установщик повторно,
и часть агентов уже установлена (частичная установка, кривые токены,
переустановка после починки)?
**Выбрано**: **одно меню в начале R3 с default'ом «перезаписать начисто»**.

### Причина — боль от ЦА
Наша ЦА (решение #1 в первом handoff — «бизнесмены, не-разработчики»)
не готова вводить `openclaw agents delete`, `rm -rf ~/.openclaw/...`
вручную в терминал. Любая техкоманда = барьер и риск опечаток.

Первая версия установщика показывала меню «skip / пересоздать /
прервать» **по каждому** уже существующему агенту. При повторной
установке всех трёх — три меню подряд. Плюс «пересоздать» удаляло
только агента, а workspace-папку (`~/.openclaw/workspace-<id>`),
channel account (`channels.telegram.accounts.<id>`) и auth-profile
(`~/.openclaw/agents/<id>/`) оставляло — потенциальные источники
странного поведения при повторном `agents add`.

### Решение
1. В R3 **одним проходом** собираем список: кто из `{tech, marketer,
   producer}` уже есть.
2. Если никого не нашли — молча идём дальше (свежая установка).
3. Если нашли — **одно** меню:
   ```
   1) Перезаписать начисто (рекомендуется, default на Enter)
   2) Пропустить уже установленных
   3) Прервать
   ```
4. При `--config` или `--install` без TTY — выбираем **1** автоматически
   (клиент явно сказал «ставь», не надо прерываться на меню).

### Новая функция `cleanup_agent_completely(agent_id)` в `lib/agents.sh`
Идемпотентный full-clean одного агента. Убирает **всё**, что
установщик когда-либо для него создавал:
1. `openclaw agents delete <id> --yes`
2. `openclaw channels remove --channel telegram --account <id> --yes`
3. `rm -rf ~/.openclaw/workspace-<id>`
4. `rm -rf ~/.openclaw/agents/<id>`

Все команды с `|| true`. Если чего-то уже нет — не ошибка.

### Что НЕ решает это
- **Контент** в `MEMORY.md` и `USER.md` теряется при overwrite. Это
  осознанно: на MVP-уровне клиент ещё не нарабатывал долгую историю.
  Когда накопится реальная обратная связь — возможно wave'ом добавим
  `--update-templates-only`, который обновляет только IDENTITY+AGENTS,
  а MEMORY/USER не трогает.
- **Токены** всё равно просим заново. В памяти между запусками их не
  храним — это security. Клиент копипастит три токена ещё раз, это
  30 секунд.

**Пересматривать**: если поступит жалоба «потерял свой MEMORY.md после
переустановки» — делаем `--update-templates-only`. До тех пор не трогаем.

## Решение #2. Проверка duplicate-bot на этапе ввода токенов

**Дата**: 2026-04-19 (по тому же bug-репорту)
**Вопрос**: что делать, если клиент случайно вставит один и тот же
токен для двух разных агентов (вставил токен Продюсера для Маркетолога)?

**Выбрано**: **проверка username'а бота через `getMe` на уникальность
против уже введённых ранее в той же сессии**.

### Причина
Первая версия валидировала только что токен рабочий (через Telegram
`getMe` API). Если клиент случайно вставил токен от другого бота —
проверка проходила: «ок, токен рабочий, бот есть». И сохраняли.
Итог: два агента привязаны к одному Telegram-боту → роутинг
ломается, третий агент вообще без токена.

### Решение
После валидации через `getMe` делаем **дополнительную** проверку:

```bash
already_used_for=""
for prev_agent in "${AGENTS_TO_INSTALL[@]}"; do
  [[ "$prev_agent" == "$agent" ]] && break  # дошли до текущего
  prev_var="BOT_USERNAME_$prev_agent"
  if [[ "${!prev_var:-}" == "$username" ]]; then
    already_used_for="$prev_agent"
    break
  fi
done
if [[ -n "$already_used_for" ]]; then
  warn "Бот @${username} уже указан для агента '${already_used_for}'."
  # подсказать создать нового через @BotFather, спросить retry
  continue
fi
```

Если username-коллизия — подсказываем **с кем именно** конфликт и
даём retry через `continue` (согласно решению #20 в первом handoff).

### Чего это НЕ покрывает
- Если клиент создаст трёх ботов с **одинаковыми** именами в разных
  BotFather-аккаунтах — не наш случай (у одного клиента один BotFather).
- Если у клиента через @BotFather осознанно есть бот с
  перепутанным username (например, `prodsrdtvbot` для технаря) — мы не
  узнаем, что username не соответствует содержанию. Это уже не наше
  дело: если клиент сам назвал бота неправильно — он и будет знать.

**Пересматривать**: нет.

## Где хранятся общие bash-правила

Применимы ко **всем** установщикам Антона (и этому, и первому):

- **Правило #19** (первый handoff, decisions-log): _bash-4+ фичи только
  через version-gate в первых 20 строках_
- **Правило #20** (первый handoff, decisions-log): _retry через
  `continue` в явном цикле, не через `exit`_

Когда появятся новые общие bash-правила — добавляем их туда. Здесь
только то, что специфично именно для agent-pack.
